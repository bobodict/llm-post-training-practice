# Interview Q&A

## Q1: 这个项目的核心问题是什么？

研究小规模中文领域数据进行后训练时，任务适应能力和通用能力保持之间的权衡。项目用 Base、General-SFT、Domain-SFT 和 Domain-DPO 做对照，并把训练指标和评测结果结构化保存。

## Q2: 为什么使用 LoRA？

LoRA 冻结基座模型，只训练低秩更新矩阵。它显著降低显存和优化器状态开销，适合在单卡环境验证数据处理和后训练算法。代价是适配器容量有限，任务数据过少时仍然容易过拟合。

## Q3: SFT 中为什么要 mask padding？

固定长度 padding 只是为了组成 batch，不应成为监督目标。将 attention mask 为 0 的位置改为 `-100` 后，PyTorch 的交叉熵会忽略这些位置，loss 更接近有效 token 的真实训练信号。

## Q4: DPO 的 reference 为什么使用 SFT 模型？

policy 从 SFT 模型继续训练时，使用 SFT 模型作为 reference 可以让偏好优化约束在已经获得指令能力的策略附近。若使用 base model，reference 与 policy 的起点不同，可能放大不必要的分布偏移。本项目明确记录了这个选择。

## Q5: 项目是否完整实现了 GRPO？

没有把它包装成完整的大规模 GRPO。项目已经实现并实际运行了单卡 smoke：真实 rollout、可验证 reward、组内标准化、PPO-style clipped policy loss、LoRA 更新和 checkpoint；但还缺少生产级 KL 约束、并行 rollout、长序列优化、多随机种子和大规模评测。

## Q6: 小数据集实验有什么局限？

当前数据是人工构造的演示语料，只有几十条样本和少量 preference pairs，没有统计意义上的测试集。因此实验主要证明代码链路和分析方法成立，不能证明模型在真实分布上的泛化能力。后续需要扩大数据、独立留出测试集，并做多随机种子实验。

## Q7: 如果结果变差，如何定位？

先区分数据问题、训练问题和评测问题：检查训练/验证 loss 是否分离，确认 padding mask 和 response mask 正确，再看每个任务的混淆矩阵和 bad case。若训练集变好而验证集变差，优先检查数据规模、epoch、学习率和偏好对质量，并考虑早停或混合通用数据。

## Q8: 下一步研究计划是什么？

先扩充并清洗中文短文本数据，建立独立测试集；再比较 SFT、DPO 和 GRPO 在同一 reward 定义下的差异，记录多次运行的均值与方差；最后补充推理吞吐、显存和量化实验，形成算法效果与系统代价的联合分析。
